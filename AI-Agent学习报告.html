PART 01

第一部分：Agent 生产环境实战——十大常见错误

入门第一课 · 建立生产环境直觉
▼
来源：Agent编排的10个常见错误：生产环境踩坑总结 · 腾讯云开发者社区 · 2026-05-16

Agent智能体 — 能自主规划、调用工具并完成目标的 AI 系统 编排在 Demo 阶段看起来很简单：A 节点做完做 B 节点，B 节点做完做 C 节点。但一旦进入生产环境，并发、超时、失败、重试、降级、Observability可观测性 — 追踪系统运行状态的能力……每一个环节都可能出问题。以下总结生产环境中 10 个真实踩坑及解决方案。

#01

坑 1：串行执行所有任务，效率极低

错误表现

需要同时获取用户信息、订单记录、商品详情。开发者写成了：先查用户 → 拿到结果后再查订单 → 拿到结果后再查商品。三个任务是顺序执行的。

问题分析

这三个任务之间没有依赖关系，完全可以并行执行。顺序执行的总耗时 = 任务1 + 任务2 + 任务3。

解决方案

使用并行节点，将无依赖的任务同时执行。并行执行的总耗时 = max(任务1, 任务2, 任务3)。

产品经理视角：用户感知的响应速度直接决定留存——无依赖的数据查询必须并行，否则产品 SLA 无法达标。
#02

坑 2：没有设置超时，节点永远挂起

错误表现

某个节点调用外部 API，外部服务挂了，请求一直没有返回。整个工作流卡在这个节点上，直到超时（默认可能是几分钟甚至无限）。

问题分析

外部服务不可靠。没有超时保护的节点，可能成为整个工作流的阻塞点。

解决方案

每个节点设置合理的超时时间（如 30 秒、60 秒）；超时后进入异常处理分支，而不是无限等待；超时阈值根据节点特性设置（调用 LLM 可设长一些，调用内部 API 设短一些）。

产品经理视角：超时不是技术细节，是用户体验底线——必须在 PRD 中定义各环节的响应时间上限。
#03

坑 3：忽略重试机制，瞬时错误直接失败

错误表现

网络抖动、限流、服务瞬时不稳，这些本可以自动恢复的错误，因为没有重试机制直接导致节点失败，整个工作流中断。

问题分析

生产环境中的失败很多是瞬时性的，自动重试可以解决大部分。

解决方案

为节点配置自动重试策略（如最多 3 次，间隔 1 秒、2 秒、4 秒）；区分可重试错误（限流、超时、网络）和不可重试错误（参数错误、权限不足）；重试次数过多时触发告警。

产品经理视角：区分「可恢复失败」与「不可恢复失败」，决定产品是静默重试还是向用户展示错误。
#04

坑 4：条件分支缺失默认兜底

错误表现

分支条件写了：如果是 A 类问题走分支 A，B 类问题走分支 B。结果来了一个 C 类问题，没有匹配到任何分支，工作流直接中断。

问题分析

基于枚举值的分支，天然存在「枚举不全」的风险。业务变化后可能新增类型。

解决方案

每个条件节点增加「默认/兜底」分支；兜底分支可以走人工处理或记录异常并降级；兜底触发时记录日志，便于发现枚举遗漏。

产品经理视角：业务场景永远在变化——兜底分支是产品韧性的最低保障，不能省略。
#05

坑 5：上下文越传越大，Token 爆炸

错误表现

节点 A 的输出传给节点 B，节点 B 把自己的输出也追加到上下文中，节点 C 再追加……处理 10 个节点后，上下文里塞了几万 Token。

问题分析

链式传递导致上下文不断膨胀。后续节点可能不需要前面的全部信息。

解决方案

明确每个节点需要的上下文范围；设置上下文最大长度（如 64K），超限时截断或摘要；关键信息提取后传给下游，而非原始数据。

产品经理视角：Token 成本与延迟是产品的隐性 KPI——上下文管理直接影响单次交互成本和响应速度。
#06

坑 6：没有熔断，雪崩效应

错误表现

下游模型服务开始变慢，请求积压。上游还在持续发起新请求，导致整个系统响应越来越慢，最终全部超时。

问题分析

没有熔断机制时，一个服务的故障会级联放大，拖垮整个系统。

解决方案

设置熔断阈值（如连续失败 5 次或错误率超 50%）；熔断后快速失败，不再发请求到下游；熔断半开后逐步恢复，验证下游已正常。

产品经理视角：熔断保护的是整体可用性——产品需要在「部分功能降级」与「全站崩溃」之间做取舍。
#07

坑 7：节点粒度太粗，难以复用

错误表现

一个节点做了太多事情：调用 LLM、解析结果、调用 API、写数据库。功能耦合在一起，其他场景想复用其中一部分都做不到。

问题分析

节点粒度是设计问题，不是技术问题。粗粒度节点灵活性和复用性都差。

解决方案

每个节点只做一件事；节点按能力类型拆分：LLM 调用节点、数据查询节点、结果解析节点、存储节点；细粒度节点可以组装成复杂工作流，也可以单独使用。

产品经理视角：模块化设计让产品迭代更快——新场景可以复用已有节点而非从零搭建。
#08

坑 8：并发资源不加控制，打爆服务

错误表现

并行节点同时调用了 20 个 LLM 请求，每个请求消耗大量显存和 GPU 资源，导致模型服务 OOM 或响应极慢。

问题分析

并行是好事，但并行度需要控制。资源是有限的。

解决方案

设置并行节点的最大并发数（如最多 5 个同时执行）；使用队列缓冲超出并发限制的请求；对高成本节点（如 LLM 调用）单独设置限流。

产品经理视角：并发上限需要在成本预算和用户体验之间找到平衡点。
#09

坑 9：错误处理分支缺失，失败后无法恢复

错误表现

正常流程写得很完整，但一旦某个节点失败，没有定义「失败了怎么办」。工作流直接中断，需要人工介入才能恢复。

问题分析

生产环境的失败是必然的。没有错误处理的设计，是不完整的设计。

解决方案

每个可能失败的节点都配置异常分支；异常处理动作：重试、降级、跳过、人工介入；关键节点失败时发送告警通知。

产品经理视角：每个失败路径都应有产品层面的「Plan B」——转人工、降级回答或引导用户重试。
#10

坑 10：没有可观测性，出问题无从查起

错误表现

工作流执行失败了，只知道「失败了」。不知道哪个节点失败的、输入是什么、错误是什么、耗时多久。

问题分析

Agent 编排是一个多节点、多分支的复杂执行过程。没有可观测性，就像在黑盒里调试。

解决方案

每个节点记录：开始时间、结束时间、输入、输出、错误信息；全链路追踪：一次工作流执行生成唯一 Trace ID，贯穿所有节点；可视化执行日志，支持按 Trace ID 检索和回放。

产品经理视角：可观测性是产品迭代的数据基础——没有 Trace，就无法定位用户投诉的根因。

避坑对照表

常见错误解决方案
串行执行无依赖任务改为并行
无超时每个节点设置超时 + 异常分支
无视重试配置自动重试策略
分支无兜底增加默认分支
上下文膨胀截断或摘要，控制长度
无熔断配置熔断阈值和恢复策略
节点粒度过粗按能力拆分为细粒度节点
并发无控制设置最大并发数 + 队列
无错误处理每个节点配置异常分支
无可观测性全链路追踪 + 日志记录
核心收获
  • Demo 能跑 ≠ 生产可用：并发、超时、重试、熔断、可观测性是上线前的必选项
  • 节点设计应遵循单一职责，便于复用和独立调试
  • 上下文和 Token 管理是成本和性能的双重杠杆
  • 每个分支和失败路径都需要产品层面的兜底策略
  • Trace ID 全链路追踪是定位生产问题的第一工具
PART 02

第二部分：Agent 工程行业全景——LangChain 2026 年度报告

了解行业全貌 · 市场在哪里、挑战是什么
▼
来源：State of Agent Engineering · LangChain · 2026

引言

进入 2026 年，组织不再问「是否要做 Agent」，而是「如何可靠、高效、规模化地部署 Agent」。LangChain 调研了超过 1,340 名专业人士（工程师、产品经理、业务领导与高管），揭示 AI Agent 的使用现状与挑战。

核心发现速览

  • 生产动量真实存在：57.3% 受访者已有 Agent 在生产环境运行；大型企业领先采用
  • 质量是生产杀手：32% 将质量列为首要障碍；成本担忧较去年下降
  • 可观测性已成标配：89% 已实现 Agent 可观测性，远超评估（Evals）采用的 52.4%
  • 多模型是常态：OpenAI GPT 领先，但 Gemini、Claude、开源模型均有显著采用；Fine-tuning（微调）尚未广泛普及

什么是 Agent Engineering（Agent 工程）？

Agent EngineeringAgent 工程 — 将 LLM 迭代打磨为可靠系统的工程过程 是一个迭代过程：由于 Agent 具有Non-deterministic非确定性 — 相同输入可能产生不同输出，工程师需要快速迭代以持续改进 Agent 质量。

生产部署现状与企业规模差异

超过半数受访者（57.3%）已有 Agent 在生产环境运行，另有 30.4% 正在积极开发并计划部署。相比去年调研的 51% 生产部署率，组织正从 PoC 阶段迈向生产。

规模差异（文字描述图表数据）：

  • 10,000+ 人企业：生产部署 67%，积极开发中 24%
  • <100 人企业：生产部署 50%，积极开发中 36%

大型企业更快从试点走向持久系统，可能得益于平台团队、安全与可靠性基础设施的更大投入。

主流应用场景排名

受访者只能选择一项主要用例，数据如下：

  • 客户服务（Customer Service）：26.5% — 最常见，表明团队正将 Agent 直接面向客户
  • 研究与数据分析（Research & Data Analysis）：24.4%
  • 内部工作流自动化：18% — 提升员工效率
  • 其他用例（代码生成、内容创作等）分布更分散，表明 Agent 应用正在多元化

10,000+ 人企业差异：内部生产力居首（26.8%），客户服务（24.7%）与研究分析（22.2%）紧随其后——大企业可能优先驱动内部效率，再同步面向终端用户部署。

最大落地障碍

  • 质量（Quality）— 首要障碍，32%：涵盖准确性、相关性、一致性，以及维持正确语气、遵守品牌/政策指南的能力
  • 延迟（Latency）— 第二大，20%：Agent 进入客户服务、代码生成等面向客户场景时，响应时间成为 UX 关键；更强多步 Agent 质量更高但响应更慢
  • 成本：较去年更少被提及——模型价格下降与效率提升转移了关注焦点

2,000+ 人企业差异：质量仍居首，但安全（Security）跃升为第二（24.9%），超过延迟。10,000+ 人企业的开放式回答指出：幻觉（Hallucination）、输出一致性、Context Engineering上下文工程 — 策划和维护 LLM 推理上下文 与大规模上下文管理是质量保障的最大挑战。

可观测性实践

追踪多步推理链与工具调用的能力已成为 Agent 的「标配」：

  • 89% 组织已实现某种形式的可观测性
  • 62% 具备详细追踪，可检查单个 Agent 步骤与工具调用
  • 已有生产 Agent 的组织：可观测性 94%，完整追踪 71.5%

没有 Agent 推理与行动的可视性，团队无法可靠调试失败、优化性能或建立内外部信任。

评估与测试实践

  • 离线评估（Offline Evals）：52.4% 在测试集上运行
  • 在线评估（Online Evals）：37.3%，正在增长
  • 生产 Agent 组织：「不评估」比例从 29.5% 降至 22.8%；在线评估 44.8%
  • 运行评估的组织中，近 25% 同时采用离线与在线评估
  • 人工审查（Human Review）：59.8%
  • LLM-as-Judge：53.3%
  • 传统 ML 指标（ROUGE/BLEU）：采用有限，不太适合开放式 Agent 交互

模型与工具生态格局

  • 超过 2/3 组织使用 OpenAI GPT 模型
  • 超过 3/4 在生产或开发中使用多个模型——按复杂度、成本、延迟路由任务
  • 约 1/3 组织自托管模型（成本优化、数据驻留、监管约束）
  • Fine-tuning（微调）：57% 未做微调，依赖基础模型 + Prompt Engineering + RAG检索增强生成 — 从外部知识库检索信息增强 LLM 回答；微调需大量数据、标注、训练基础设施，保留给高影响/专业化场景

日常使用的 Agent 类型（开放式回答归纳）

  1. 编码 Agent 主导：Claude Code、Cursor、GitHub Copilot、Amazon Q、Windsurf 等——代码生成、调试、测试、导航代码库
  2. 研究与深度研究 Agent：ChatGPT、Claude、Gemini、Perplexity——探索新领域、总结长文档、跨源综合
  3. 基于 LangChain/LangGraph 的自定义 Agent：QA 测试、内部知识库搜索、SQL/Text-to-SQL、需求规划、客户支持、工作流自动化
  4. 仍有相当比例用户尚未超越 LLM 聊天或编码辅助——「全面 Agent 化」仍处于早期

调研方法论

  • 时间：2025 年 11 月 18 日 – 12 月 2 日，公开调研 2 周
  • 样本量：1,340 份有效回复
  • 行业 Top 5：科技 63%、金融 10%、医疗 6%、教育 4%、消费品/制造各 3%
  • 公司规模：<100 人 49%、100-500 18%、500-2000 15%、2000-10000 9%、10000+ 9%
核心收获
  • Agent 生产部署已过临界点（57%+），问题从「做不做」转向「怎么做好」
  • 质量 > 延迟 > 安全（大企业）是落地优先级排序
  • 可观测性（89%）远领先于评估（52%）——「能看见」比「能量化」更成熟
  • 多模型路由已成标准实践，微调仍是少数派选择
  • 客户服务与研究分析是当前两大主战场，编码 Agent 是开发者日常最高频工具
PART 03

第三部分：Agent 评估体系——Amazon 实战经验

系统评估 Agent 质量 · 建立评估思维
▼
来源：Evaluating AI agents: Real-world lessons from Amazon · AWS Machine Learning Blog

生成式 AI 行业已从 LLM 驱动应用演进为 Agentic AI智能体 AI — 能自主规划、调用工具、迭代解决问题的 AI 系统。Amazon 自 2025 年起已在各组织构建了数千个 Agent。单一模型 Benchmark 不足以评估 Agent——新范式需评估工具选择准确性、多步推理连贯性、记忆检索效率及任务完成率等系统涌现行为。

Amazon Agent 评估框架概览

框架包含两大组件：自动化 Agent 评估工作流 + Agent 评估库（基于 Amazon Bedrock AgentCore Evaluations）。

四步评估工作流

  1. 定义评估输入：通常是 Agent 执行的 Trace 文件——离线 Trace（任务完成后上传）或在线 Trace（定义评估维度与指标）
  2. 自动生成评估指标：评估库生成默认与用户自定义指标
  3. 共享评估结果：通过 S3 存储桶或 Dashboard 可视化 Trace 可观测性与评估结果
  4. 分析与监控：Agent 性能审计；定义规则在性能下降时发送通知；HITLHuman-in-the-Loop（人在回路） — 人工参与审核与校准 机制定期审计 Trace 子集

三层评估架构

底层（Bottom Layer）：基础模型评估

对多个 Foundation Model 进行 Benchmark，选择驱动 Agent 的合适模型，衡量不同模型对 Agent 整体质量与延迟的影响。

中间层（Middle Layer）：组件评估

评估 Agent 各组件性能：意图检测、多轮对话、记忆、LLM 推理与规划、工具使用等。例如：Agent 是否正确理解用户意图？LLM 如何通过 CoTChain-of-Thought（思维链） — 逐步推理的提示策略 驱动工作流规划？工具选择与执行是否与计划一致？

上层（Upper Layer）：任务完成评估

评估 Agent 最终响应、任务完成度、是否达成用例目标；涵盖整体责任与安全、成本及客户体验影响。

评估指标定义（完整）

最终响应质量（Final Response Quality）

  • Correctness（正确性）：AI 助手响应的事实准确性与正确性
  • Faithfulness（忠实度）：响应是否与对话历史保持一致
  • Helpfulness（有用性）：响应如何有效帮助用户推进目标
  • Response Relevance（响应相关性）：响应与具体问题的匹配程度
  • Conciseness（简洁性）：信息传达效率，是否简洁且不遗漏关键信息

任务完成（Task Completion）

  • Goal Success（目标成功）：AI 助手是否在会话中成功完成所有用户目标
  • Goal Accuracy（目标准确性）：将输出与 Ground Truth 对比

工具使用（Tool Use）

  • Tool Selection Accuracy（工具选择准确性）：是否为给定情境选择了合适工具
  • Tool Parameter Accuracy（工具参数准确性）：调用工具时是否正确使用了上下文信息
  • Tool Call Error Rate（工具调用错误率）：工具调用失败的频率
  • Multi-turn Function Calling Accuracy（多轮函数调用准确性）：多工具是否被调用、调用顺序是否正确

记忆（Memory）

  • Context Retrieval（上下文检索）：评估从记忆中检索相关上下文的准确性，平衡精确率与召回率

多轮对话（Multi-turn）

  • Topic Adherence Classification（主题 adherence 分类）：多主题对话中是否保持在预定义领域
  • Topic Adherence Refusal（主题 adherence 拒绝）：Agent 是否拒绝回答特定主题的问题

推理（Reasoning）

  • Grounding Accuracy（ grounding 准确性）：模型是否理解任务、恰当选择工具，CoT 是否与上下文及工具返回数据一致
  • Faithfulness Score（忠实度分数）：推理过程的逻辑一致性
  • Context Score（上下文分数）：Agent 每一步是否有上下文依据

责任与安全（Responsibility and Safety）

  • Hallucination（幻觉）：输出是否与已知知识一致，是否包含虚构/误导内容
  • Toxicity（毒性）：是否包含有害、冒犯、歧视性语言
  • Harmfulness（危害性）：是否包含侮辱、仇恨言论、暴力、不当性内容、刻板印象等

案例一：Amazon 购物助手——工具使用评估

购物助手需与数百甚至数千个底层 API/Web 服务交互，提供客户画像、产品发现、库存查询、下单等个性化体验。手动接入 API 通常需数月。

挑战：工具 Schema 与语义描述不精确 → 运行时错误工具选择 → 无关 API 调用 → 上下文膨胀 → 延迟与成本上升。

解决方案：

  • 建立跨组织工具 Schema 与描述标准化治理框架
  • LLM 驱动的 API 自助接入系统，自动生成标准化工具 Schema 与描述
  • 从历史 API 调用日志合成 Golden Dataset 用于回归测试
  • 用工具选择准确性、参数准确性、多轮函数调用准确性系统性评估

案例二：Amazon 客服 Agent——意图识别评估

编排 Agent 使用推理模型检测客户意图，决定路由到哪个专用 Resolver（工具或 Sub-agent）。意图识别错误 → 错误路由 → 无关回复 → 客户挫败 → 转人工成本上升。

评估方法：

  • LLM Simulator：LLM 驱动的虚拟客户 Persona 模拟多样用户场景
  • 模拟数据集：匿名历史交互中的用户查询 + Ground Truth 意图对
  • 对比 Agent 生成意图与 Ground Truth，验证意图正确性
  • 同时评估任务完成（最终响应与意图解决）及多轮指标（主题 adherence 分类与拒绝）

案例三：Amazon 多 Agent 系统——卖家助手评估

工作流：LLM Planner 接收请求 → 分解子任务 → 分配给专用 Sub-agent → Sub-agent 自主执行 → 回报编排 Agent → 聚合输出。

评估维度：

  • 各专用 Agent 的任务完成、推理、工具使用、记忆检索
  • Planning Score（规划分数）：子任务是否成功分配给 Sub-agent
  • Communication Score（通信分数）：Sub-agent 间通信消息质量
  • Collaboration Success Rate（协作成功率）：子任务成功完成百分比
  • HITL 评估 Agent 间通信、任务分解合理性、冲突解决策略、多 Agent 决策逻辑一致性

经验教训与最佳实践

  1. 多维度 holistic 评估：超越传统准确率，涵盖质量（推理连贯性、工具选择、任务完成）、性能（延迟、吞吐、资源利用）、责任（安全、毒性、偏见、幻觉）、成本（推理、工具调用、人工纠错）
  2. 用例特定评估：如客服需客户满意度、首次解决率、情感分析；需与领域专家协作定义成功标准与 Golden Dataset
  3. HITL 是关键组件：评估推理链、多步工作流连贯性、与业务需求对齐；提供 Ground Truth 标签；校准 LLM-as-Judge 与人工偏好
  4. 生产环境持续评估：预部署评估无法完全捕获生产特性；监控真实用户行为、使用模式与边缘案例；追踪关键指标、告警阈值、异常检测；发现问题后模型重训、上下文工程优化
核心收获
  • Agent 评估必须三层递进：模型 → 组件 → 任务完成
  • 工具 Schema 标准化是大规模 Agent 集成的先决条件
  • LLM Simulator + Golden Dataset 是意图识别评估的有效范式
  • 多 Agent 系统需要额外的协作指标（规划、通信、协作成功率）
  • HITL 不是可选项，而是高 stakes 场景的质量保障
PART 04

第四部分：上下文工程——Anthropic 官方指南

从 Prompt Engineering 进阶到 Context Engineering
▼
来源：Effective context engineering for AI agents · Anthropic Engineering · 2025-09-29

上下文（Context）是 AI Agent 的关键但有限资源。本文探讨如何有效策划和管理驱动 Agent 的上下文。

Context Engineering vs. Prompt Engineering

在 Anthropic 看来，Context Engineering上下文工程 — 策划和维护 LLM 推理时最优 Token 集合的策略 是 Prompt Engineering提示词工程 — 编写和组织 LLM 指令以获得最佳结果 的自然演进。

  • Prompt Engineering：聚焦如何编写有效 Prompt，尤其是 System Prompt——适用于一次性分类或文本生成
  • Context Engineering：策划和维护 LLM 推理期间的最优 Token 集合——包括 System Instructions、Tools、MCPModel Context Protocol（模型上下文协议） — 标准化 AI 与外部工具/数据交互的协议、外部数据、消息历史等

Agent 在循环中运行，不断产生可能与下一轮推理相关的数据，必须周期性精炼。Context Engineering 是策划「从不断演变的信息宇宙中选择什么放入有限上下文窗口」的艺术与科学——每次决定传递给模型什么时都会发生。

为什么 Context Engineering 对 Agent 至关重要

尽管 LLM 速度越来越快、能处理越来越大的数据量，但我们观察到 LLM 像人类一样，在某一节点会失焦或混乱——这就是 Context Rot上下文腐化 — 上下文 Token 增多时模型准确回忆信息能力下降。

  • 上下文必须被视为具有递减边际效用的有限资源
  • LLM 有「注意力预算（Attention Budget）」——每个新 Token 都会消耗预算
  • Transformer 架构使每个 Token 关注整个上下文，产生 n² 成对关系
  • 上下文越长，捕获这些关系的能力越 stretched——上下文大小与注意力焦点之间存在自然张力
  • Position Encoding Interpolation 等技术允许更长序列，但 Token 位置理解会有所退化

有效上下文的组成

System Prompt（系统提示词）

应极其清晰，使用简单直接的语言，在合适的「高度（Altitude）」呈现想法：

  • 过高（失败模式 1）：硬编码复杂 brittle 逻辑——创造脆弱性
  • 过低（失败模式 2）：模糊的高层指导——未给 LLM 具体信号
  • 最优：足够具体以有效引导行为，又足够灵活以提供强启发式

建议组织为 distinct sections（<background_information>、Tool Guidance、Output Description 等），用 XML 标签或 Markdown 标题分隔。追求最小但完整的信息集——minimal 不等于 short。

Tools（工具）

工具定义 Agent 与信息/行动空间的契约。好工具应：

  • 返回 Token 高效的信息
  • 鼓励高效的 Agent 行为
  • 自包含、健壮、意图清晰——类似设计良好的代码库函数
  • 避免 bloated tool sets——若人类工程师无法确定该用哪个工具，Agent 更不行

Examples（示例 / Few-shot）

继续强烈建议 Few-shot Prompting，但不要在 Prompt 中堆砌 edge case 规则清单。应策划一组 diverse、canonical 的示例，有效展示 Agent 预期行为——对 LLM 而言，示例是「一图胜千言」。

Message History（消息历史）

总体指导： thoughtful 且 tight——信息丰富但精简。

上下文检索与 Agent 搜索策略

Anthropic 对 Agent 的定义：LLM 在循环中自主使用工具。

三种策略对比

策略描述适用场景
Pre-retrieval（预检索）推理前用 Embedding 检索所有相关数据内容相对静态的场景
Just-in-time（即时检索）维护轻量标识符（文件路径、查询、链接），运行时动态加载大型数据库、动态内容
Hybrid（混合策略）部分数据预加载 + Agent 自主探索Claude Code：CLAUDE.md 预加载 + glob/grep 即时检索

Just-in-time 优势：模拟人类认知——不记忆整个 corpus，而是通过文件系统、索引、书签按需检索。Metadata（文件夹层级、命名约定、时间戳）提供行为精炼信号。支持 Progressive Disclosure（渐进式披露）——Agent 逐层发现相关上下文。

Trade-off：运行时探索比预计算检索慢；Agent 可能误用工具、走死胡同。混合策略可能更适合法律、金融等动态性较低的场景。

长期任务的三种关键技术

对于跨越数分钟到数小时、Token 超过上下文窗口的任务（大型代码迁移、综合研究项目）：

1. Compaction（压缩）

对话接近上下文窗口限制时，总结内容并重新启动新上下文窗口。

  • Claude Code 实践：将消息历史传给模型总结，保留架构决策、未解决 Bug、实现细节，丢弃冗余工具输出
  • 压缩后继续：压缩上下文 + 最近访问的 5 个文件
  • 艺术在于选择保留 vs 丢弃——过度压缩会丢失后期才显现重要性的 subtle 上下文
  • 最轻量的压缩形式：Tool Result Clearing——清除历史中的原始工具调用结果

2. Structured Note-taking（结构化笔记 / Agentic Memory）

Agent 定期将笔记持久化到上下文窗口外的 Memory，后续再拉回上下文。

  • Claude Code 创建 to-do list；自定义 Agent 维护 NOTES.md
  • Claude 打 Pokémon 案例：Agent 在数千步游戏中维护精确计数（「过去 1,234 步在 Route 1 训练，Pikachu 升了 8 级，目标 10 级」）；自发开发区域地图、成就记录、战斗策略笔记；上下文重置后读取笔记继续多小时训练——实现单上下文窗口不可能的长程策略
  • Sonnet 4.5 发布的 Memory Tool：基于文件的 Memory 系统，Agent 可跨 Session 积累知识库

3. Sub-agent Architectures（子 Agent 架构）

专用 Sub-agent 处理聚焦任务，各自拥有干净的上下文窗口。主 Agent 协调高层计划，Sub-agent 做深度技术工作或工具检索。

  • Sub-agent 可能探索数万 Token，但只返回 1,000-2,000 Token 的 condensed summary
  • 详细搜索上下文隔离在 Sub-agent 内，主 Agent 聚焦综合与分析
  • 多 Agent 研究系统显示：复杂研究任务上 Substantial Improvement over single-agent

选择指南：Compaction 适合需要大量 back-and-forth 的任务；Note-taking 适合有清晰里程碑的迭代开发；Multi-agent 适合复杂研究与分析（并行探索收益大）。

Claude Code 实践案例

Claude Code 是混合策略的典型：CLAUDE.md 文件 upfront 加载到上下文；glob、grep 等原语允许 Just-in-time 导航环境、检索文件——绕过 stale indexing 和复杂语法树问题。对大型数据库执行复杂数据分析：模型编写 targeted query、存储结果、用 head/tail 分析大量数据——无需将完整数据对象加载到上下文。

核心收获
  • Context Engineering 是 Prompt Engineering 在 Agent 时代的自然演进——管理整个上下文状态而非仅 System Prompt
  • 上下文是有限资源：Attention Budget 随 Token 增加而递减
  • System Prompt 追求「最小但完整」；Tools 应精简无重叠；Examples 重质不重量
  • 检索策略：Pre-retrieval / Just-in-time / Hybrid 三选一或组合
  • 长期任务三板斧：Compaction、Structured Notes、Sub-agent Architecture

第五部分：AI 原生工作方法论（日常随查手册）

日常 AI 协作操作指南 · 贯穿所有实践环节
▼
来源：AI 原生工作报告（PDF）· 十个关键词结构

研究团队通过十个关键词，系统梳理 AI 协作的方法论。以下为完整核心观点与行动建议。

序：信任鸿沟

核心观点：可靠不等于不出错，可靠是出错之后还能接住。

行动建议
  • 不追求 AI 零出错。每次犯错，记下错误类型和修复方式。经历过错误并被解释清楚的信任，比从未出错的信任更结实。
  • 复杂任务人要留在驾驶座上，简单任务可放手让跑。注意保留了多少主动检查环节，控制粒度要匹配任务复杂度。
  • 一号位带头打破「不愿告诉同事自己在用 AI」的沉默。坦诚聊哪里用得顺、哪里踩坑。信任在透明对话中生长。
与 Agent 开发的关联：Agent 产品的容错设计（Graceful Degradation）和 HITL 机制，是建立用户信任的产品基础设施。

关键词一：驾驭工程

核心观点：人设计环境，AI 在环境中执行——核心范式跃迁。同一个模型，换一套基础设施，表现天差地别。

行动建议
  • 每次 AI 犯错，花时间把修复方案工程化，让它再也不犯同样的错。
  • 对 AI 产出不满意时，克制直接改的冲动——问：能不能写一条规则、建一个模板、加一道检查来规避？
  • 对照四象限（前馈/反馈 × 确定性/推理性）标出已有与缺失。大多数人只做了前馈（说清楚要求），反馈（做完后自动检验）才是更高性价比的优化方向。
与 Agent 开发的关联：Agent 编排中的节点设计、重试策略、评估 Gate 就是「驾驭工程」——把一次性的 Prompt 修复转化为持久的工作流规则。

关键词二：记忆

核心观点：精确遗忘比完美记忆更重要。核心挑战不只是怎么记住更多，还要忘得更精确。

行动建议
  • 在工作空间放一个 200 行以内的记忆起点文件（markdown），声明核心约定、常见错误、偏好设置。每周看一眼，超了再优化结构。
  • 记忆系统最容易犯的错是只写入不管理。每隔一两周审查：哪些过时了？哪些重复了？哪些可以合并？
  • 建立关联：这个决策和那次失败有什么因果关系？碎片连成网络之后，才能推理、迁移、产生新洞见。
与 Agent 开发的关联：Agent Memory 架构（短期上下文 + 长期持久化 + 定期 Compaction）直接对应此关键词——精确遗忘是 Context Engineering 的核心技能。

关键词三：技能

核心观点：最好用的，是自己持续维护的 Skills。成熟的 Skill 会越来越短，最终留下：精准描述、领域知识、踩坑记录。

行动建议
  • 自己写一个 Skill，不超过 50 行。提炼最常重复教 AI 的做事方式：精准触发描述、AI 学不到的领域知识、从失败中总结的踩坑记录。
  • 定期清理合并 Skills。单个 Skill 超过 500 字，大概率六成不必要。模型已经会做的指令直接删。
  • 不要写显而易见的东西。聚焦能推动 AI 超越默认思路的信息。
与 Agent 开发的关联：Cursor Skills / Agent Skills 是 Agent 项目的「可复用 SOP」——把团队最佳实践工程化，减少每次从零 Prompt 的成本。

关键词四：评估

核心观点：可靠的评估与反馈是方向盘，是质量的前提。AI 的自评比人更严重——别等它学会自省，给它一个独立的裁判。

行动建议
  • 给 AI 加一个独立审查者。同一个模型，开一个新对话窗口就够。分离「做事的上下文」和「判断的上下文」。
  • 回顾最近 AI 出过的 3 个错，变成检查项。下次类似任务完成后跑一遍——这就是质量回归测试 v0.1。
  • 发现错误时，别只说「错了」。把「为什么错」和「怎么改」一起写进去——每次违规都是一次微型培训。
与 Agent 开发的关联：对应 Amazon 三层评估框架和 LangChain 报告的 Evals 实践——离线 Golden Dataset + LLM-as-Judge + HITL 是 Agent 质量保障的标准组合。

关键词五：上下文管理

核心观点：停止把你的人生故事放进去。信息太少让 AI 发散，信息太多也让 AI 犯蠢——它太听话了，把所有指令都当回事。

行动建议
  • 首轮输入只放三样东西：一句话任务目标、关键约定、非标准流程。其余按需调用。
  • 别害怕清空对话重新开始。进度笔记存入文件后，清空往往比死撑压缩更高效。对话太长后 AI 会重复旧模式、忽略新指令。
  • 过时内容留在上下文里是在主动毒化 AI 的判断。每隔一段时间审查规则和参考材料——删掉比添加更重要。
与 Agent 开发的关联：Anthropic Context Engineering 的核心原则：最小高信号 Token 集。Agent 每个节点的上下文传递范围必须精确定义。

关键词六：工作流

核心观点：人类工程师始终是这场演出的导演。有工具不等于有工作流——真正卡住大家的是重新审视整条流程的那一步。

行动建议
  • 画出最关键的工作流，标出所有手动传递数据、上下文、状态的环节——这些「人类胶水」位置往往是最大效率漏洞。
  • 检验 AI 改造是否真的在重新设计：有没有移除交接？有没有并行化？有没有把人移到异常处理？有没有增加评估循环？
  • 好的工作流是飞轮。记录哪些步骤 AI 做得好、哪些需要人介入，回注到设计中。
与 Agent 开发的关联：Agent 编排（LangGraph / Dify Workflow）的价值在于重新设计端到端流程，而非加速单个节点——产品经理应主导流程重设计。

关键词七：多智能体

核心观点：用好一个再用多个。绝大多数失败来自规则没说清楚、分工不够细。

行动建议
  • 单 Agent + 好指令 + 合适工具能搞定 90% 场景。确认真的到了瓶颈（上下文放不下、需要并行、需要故障隔离）再引入第二个。
  • 给子任务的描述要像给第一天入职的新人写工单——详细到不依赖共享上下文就能独立执行。分工要 MECE（互不重叠、完全穷尽）。
  • 一个主 Agent 编排过多执行者，协调开销会吃掉收益。从一个规划者 + 2-3 个执行者开始。信息每传递一次就衰减一次——让子 Agent 直接写文件，不通过主 Agent 转述。
与 Agent 开发的关联：Amazon 多 Agent 评估案例（Planning Score、Communication Score、Collaboration Success Rate）提供了多 Agent 架构的产品评估框架。

关键词八：加法偏见

核心观点：加法能力是智慧，减法是智慧。AI 从人类语言中训练，把这个加法偏见进行了放大。

行动建议
  • 每次想加东西之前，先问「能不能减」。减法不是删除，是策展——判断什么值得留下。
  • 审计工具集和指令库。几个月没触发的规则标记废弃。上下文窗口的注意力成本是 n² 级别的。
  • 识别哪些约束是物理限制，哪些只是过时惯例——这个判断 AI 目前做不好，是人的职责。
与 Agent 开发的关联：Agent 工具集应遵循「最小可行工具集」原则——Anthropic 和 Amazon 都强调 bloated tool sets 是 Agent 失败的主因。

关键词九：去技能化

核心观点：AI 未必让你失业，但可能先让你失去「本来会做这件事」的能力——而你可能根本不会注意到。

行动建议
  • 每隔一两周找一个小任务，完全不用 AI 做。手感在下降、犹豫在增加，那就是信号。
  • 别全盘委托。高效模式是用 AI 支撑思考，不是替代思考。把判断力和设计决策留在自己手里。
  • 保留不被打断的深度思考时间。重新练习深度阅读、独立写作、不借助工具的分析。
与 Agent 开发的关联：产品经理必须保持对 Agent 能力边界的独立判断——不能因 Agent 能跑 Demo 就认为生产可用，需保留独立评估能力。

关键词十：知识工程

核心观点：方法会过时，工具会迭代。最终留下来的是判断框架、因果链条、领域直觉。

行动建议
  • 每次 AI 出错时花 10 秒把纠正写进上下文文件，标注「这条和哪条相关」——在建连接。
  • 先写 Skill（解决重复教），再记交互痕迹（解决反复忘），积累够了连接自然涌现。给 Skill 加依赖关系，给记忆加标签，给文件加版本号。
  • 对抗知识腐烂。每次碰到过时说明，花 30 秒改掉。好的知识库不是大的，是活的。
与 Agent 开发的关联：Agent 项目的 Golden Dataset、Tool Schema 文档、评估 Rubric 就是「知识工程」的产出——是团队最不可替代的资产。
核心收获
  • AI 协作的核心范式是「驾驭工程」——人设计环境，AI 在环境中执行
  • 记忆、上下文、技能都需要主动管理——精确遗忘比完美记忆更重要
  • 评估必须独立化——分离「做事」与「判断」的上下文
  • 多 Agent 是进阶选项，90% 场景单 Agent 足够
  • 知识工程是 AI 时代个人与团队最不可替代的积累

Prompt Engineering 核心技巧与最佳实践
▼
来源：Prompt engineering overview · Anthropic Docs

开始之前的三要素

  1. 清晰定义用例的成功标准（Success Criteria）
  2. 建立实证测试方法（Evaluations）
  3. 准备第一版 Draft Prompt 用于迭代改进

何时做 Prompt Engineering vs 换模型

并非所有问题都适合 Prompt 解决。延迟和成本问题可能通过选择不同模型更容易改善。Prompt Engineering 聚焦可通过 Prompt 控制的成功标准。

核心技巧速查清单

1. 清晰直接（Be Clear and Direct）
  • 明确说明期望的输出格式、长度、风格
  • 避免模糊指令如「写得好一点」——给出具体标准
  • 把 Claude 当作聪明但没有上下文的新员工
2. 使用 XML 标签结构化
  • 用 <instructions>、<context>、<example> 等标签分隔 Prompt 各部分
  • 便于 Claude 区分指令、上下文、示例与输入数据
  • 标签名称无特殊含义，一致即可
3. 提供示例（Few-shot）
  • 2-5 个 diverse、canonical 示例优于 20 个 edge case 规则
  • 示例应展示输入→输出的完整映射
  • 示例是 LLM 理解预期行为的最有效方式
4. 让 Claude 思考（Chain of Thought）
  • 要求 Claude 在给出最终答案前先逐步推理
  • 可用 <thinking> 和 <answer> 标签分离推理与答案
  • 复杂任务显著改善准确性
5. 分配角色（Role Prompting）
  • 在 System Prompt 中定义 Claude 的角色与专业领域
  • 角色设定影响语气、深度与视角
  • 避免过于宽泛——「你是专家」不如「你是有 10 年经验的 SQL 优化工程师」
6. Prompt 链（Prompt Chaining）
  • 复杂任务拆分为多个 Prompt 步骤，前一步输出作为后一步输入
  • 每步聚焦单一子任务，降低单 Prompt 复杂度
  • 适合多阶段工作流（研究→大纲→写作→审校）
7. 预填充响应（Prefill Response）
  • 在 Assistant 消息中预填开头（如 { 强制 JSON 输出）
  • 引导输出格式，减少格式错误
8. 迭代测试流程
  • 从最小 Prompt + 最强模型开始测试
  • 基于失败模式逐步添加指令和示例
  • 每次只改一个变量，便于归因
  • 使用 Claude Console 的 Prompt Generator 和 Prompt Improver 工具
核心收获
  • Prompt Engineering 的前提是清晰的成功标准 + 可运行的 Eval
  • 结构化（XML 标签）+ 示例（Few-shot）+ 推理（CoT）是三大支柱
  • 复杂任务用 Prompt Chaining 拆分，而非堆砌单 Prompt
  • 从最小 Prompt 开始，基于失败模式迭代——不要过度设计
